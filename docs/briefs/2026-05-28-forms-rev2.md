## What we're building

Same end-state as the original brief: a definition-driven entry form renderer, three Phase 1 Tier 1 entry types (Session Log, Outreach Log, Meeting Notes), and a unified list view across them.

**New in rev 2: a schema-rework prerequisite** — before any forms code lands, a migration restructures the database to match `docs/phase1/02-schema.md`: adds `option_lists` table + seed data, adds `extras` JSONB columns, drops the auth-batch's Tier-2 detail tables + Postgres ENUMs + base `entries` table, and reshapes `members`. Two small app-code follow-ups patch the dashboard and showcase page so they still compile against the new schema.

## Why

The forms layer per `03-forms.md` is built around two architectural primitives — an `option_lists` table for user-extensible enumerations and an `extras` JSONB column on each detail table for non-queried fields. Without those, the renderer's `storage: 'column' | 'extras'` split, the "Add new…" popover, the multi-select-with-note shape, and the generic insert helper all need a different design. Going to spec is what `02-schema.md` and `03-forms.md` describe and is what every subsequent Phase 1 entry type assumes — picking the spec now means future entry types add cleanly; picking the alternative now means a bigger reckoning in Phase 2.

## Acceptance criteria

Same as rev 1 for T13–T17 (12 criteria, verbatim) — they describe the user-visible behavior, which is unchanged. **Added** prerequisite criteria:

**Schema migration (new — prerequisite for everything else)**

- `option_lists` table exists with the shape in `02-schema.md` §4.2 and the seed data in §4.3 (8 categories, ~50 seed rows).
- `session_logs`, `outreach_logs`, `meeting_notes` exist with the shape in `02-schema.md` §§5.1–5.3, each having an `extras jsonb NOT NULL DEFAULT '{}'::jsonb` column.
- The auth-batch's Postgres ENUM types (`outreach_event_type`, `meeting_type`, `permission_status`, etc.) are dropped.
- The base `entries` table + its cross-cutting children (`entry_revisions`, `entry_attendees`, `media_attachments`, `flags`, `action_items`, `awards`, `award_criteria`, `classification_tags`) are dropped. (`flags`, `classification_index`, `award_criteria_snapshot` will be re-added per spec — `flags` is a Phase 1 surface, the others are Phase 1 schema for Phase 4 use.)
- All seven Tier-2 detail tables from the auth batch (`competition_recaps`, `decision_logs`, `hardware_change_logs`, etc.) are dropped. Phase 2+ recreates them per `02-schema.md` §§5.4–5.12 when those forms ship.
- `members` shape matches `02-schema.md` §4.1 — `(id, email, display_name, created_at, updated_at, deleted_at)`. The `handle_new_auth_user()` trigger is rewritten to populate the new shape.
- `subsystems` table is dropped (spec uses `option_lists` with `category='subsystem'` instead).
- `teams`, `seasons`, `subsystem_leads` are dropped (Phase 3 multi-team is out of scope for Phase 1).
- `pnpm verify` passes; `supabase db push` against the dev project succeeds; the dev DB matches the migrations exactly.

**App-code unblock (new — keeps existing routes compiling against the new schema)**

- `app/(authed)/dashboard/page.tsx` no longer selects `name, role, team_id` from `members` (those columns don't exist in the new schema). Replaced with a select on `email, display_name` and a copy update.
- `app/showcase/page.tsx` no longer queries `entries` (that table is dropped). Either stubbed with "Showcase ships in Phase 4" copy or removed entirely (decision in the implementation PR).
- The auth-batch's RLS policies file is rewritten to apply to the new tables (`option_lists`, `session_logs`, `outreach_logs`, `meeting_notes`) with the Phase 1 permissive `FOR ALL TO authenticated USING (true)` per CLAUDE.md "Phase 3 replaces, not adds."

## Out of scope

Same as rev 1, plus:

- **Migrating data.** No production data exists; the migration is destructive on dev (DROP TABLE … CASCADE) and explicitly safe to run.
- **Multi-team / multi-role schema (Phase 3).** Spec's `members` table is single-team; the team/role concept lives in Phase 3.
- **Tier 2 detail tables.** Phase 2 recreates them per spec when those forms ship; dropping them now is fine because they're untouched by any current code path.
- **The base `entries` table.** Spec uses standalone detail tables, not a shared base. The auth batch's `entries` table is dropped without a replacement — cross-cutting reads (list view, etc.) hit each detail table directly and union in app code per `03-forms.md` §18.

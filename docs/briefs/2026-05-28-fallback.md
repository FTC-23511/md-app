## What we're building

The emergency path for capturing entries when the main app is unreachable. Three copy-paste markdown templates (`session-log.md`, `outreach-log.md`, `meeting-notes.md`) that a non-developer team member can fill out offline, hand to the Documentation Captain, and have ingested into the database via a small Node importer. The captain runs `pnpm run import-fallback -- docs/fallback/inbox/*.md` when the app is back; successful files are renamed `*.imported.md` in place, errored files get a sibling `.errors.log`. End result: an outage stops capture for the time it takes a contributor to open a text editor, not for the duration of the outage itself.

## Why

`MD_App_Charter.md` ¬ß8 names a mid-season outage during build crunch as the single biggest risk to MD's reason for existing ‚Äî preventing contemporaneous record loss. The fallback satisfies the Definition of Done item in `docs/phase1/00-plan.md` that says the app must remain capture-capable when itself is down. Selected over Google Form because the build cost is lower, no external auth dependency, and the artifacts live in the repo where future Captains can find them.

## Acceptance criteria

Observable in the browser / locally. One yes/no per task.

**T18 ‚Äî Templates and workflow docs**

- `docs/fallback/templates/` contains exactly three files: `session-log.md`, `outreach-log.md`, `meeting-notes.md`, matching the YAML+body shape spec'd in `docs/phase1/05-fallback.md` ¬ß4.
- `docs/fallback/README.md` exists and explains the contributor workflow (copy template ‚Üí fill ‚Üí send to Captain) in plain language a non-developer can follow (no git, no importer mentions in the contributor path).
- `docs/fallback/inbox/` exists in the repo via `.gitkeep`.
- Manually filling one template by hand produces a readable, valid markdown document with no parser ambiguity.

**T19 ‚Äî Importer and smoke test**

- `pnpm run import-fallback -- docs/fallback/inbox/*.md` succeeds against three pre-filled fixture files (one of each template type) checked in for the smoke test.
- After running, three new rows appear in `/list` with `created_via = 'fallback_form'`. (The App Lead verifies this in the smoke-test PR before approving.)
- The three fixture files are renamed to `*.imported.md` in place.
- Re-running the importer on the same glob results in zero new inserts (renamed files no longer match `*.md`).
- A deliberately broken fixture (missing required field) produces a `<filename>.errors.log` sibling and is _not_ inserted; the original file is unmodified.
- Option resolution: a single-select value that doesn't exist in `option_lists` is auto-inserted with `is_seed: false` when the field allows new options; rejected with a named error when it doesn't.

## Out of scope

- **AI deep-dive on Software Change Log fallback.** Phase 2.

- **Auto-link `created_by` to a member row.** Importer sets `created_by = NULL`; Captain can manually link later if desired.
- **Auto-detection of already-imported files beyond the `.imported.md` rename.** The rename + glob mismatch is the deduplication mechanism. No hash table, no DB lookup.
- **Apps Script / Google Form path.** Original T19 was a Google Form smoke test; that path is dropped per `05-fallback.md` ¬ß9 in favor of the Node script.
- **Integration tests in Vitest.** Smoke test is a human run per the answer to Open Question #3 below; CLAUDE.md absolute rule #6 says no tests in Phase 1 unless asked.
- **Any change under `supabase/migrations/`, `middleware.ts`, `app/auth/**`, or `lib/supabase/`.** Per `docs/ROUTINE.md` ¬ß4 these are approval-required surfaces, and the fallback path doesn't need them.

## Open questions

The three decisions made during planning:

1. **Sequencing.** Both T18 + T19 deferred until the Forms batch (T13‚ÄìT17) ships. T19 imports rely on `src/entries/_registry.ts`, the per-entry definitions, and `buildZodSchemaFromDefinition` from `src/lib/validate-entry.ts` ‚Äî none of which exist yet. T18 (templates + docs) could ship independently, but holding it lets both ship as a coherent Sprint D batch right after Forms lands.
2. **Dependencies.** `yaml` and `glob` are added as **devDependencies**. Both are tiny, widely-used, and live only in `scripts/fallback/` ‚Äî never imported by app code. The PR adding them must explicitly call out the addition per CLAUDE.md absolute rule #2; per `docs/ROUTINE.md` ¬ß4 these are "non-security dependency bumps" ‚Üí auto-merge tier.
3. **Smoke-test execution.** The routine ships the importer code + three pre-filled fixture files in `docs/fallback/inbox/`. The PR is approval-required. The App Lead runs `pnpm run import-fallback`, verifies three rows in `/list`, then approves and merges. No automated integration test is added.

## What we're building

The Test Log: a form that accepts **any** test's data and computes statistics automatically, with run-to-run trends. Three input modes picked by a radio at the top (`test_type`):

1. **pass/fail** — one binary outcome per trial.
2. **single measure** — one number per trial.
3. **custom** — the tester defines their own columns (name + kind) and pastes rows.

For every recognizable column shape, statistics populate without the tester typing a formula (pass rate + 95% CI, mean/stddev/min/max, category frequencies). Re-filing under the same `test_label` shows the delta on the headline stat against the prior run.

## Why

The Test Log is the data spine of the KPI dashboard and the Comp Recap companion view — it must stay queryable and trendable while accepting test shapes we can't predict. Auto-compute removes the formula-typing friction that kills capture rate. Spec: `docs/phase2/00-plan.md` step 2C; full design `docs/phase2/03-test-log.md`; tables `01-schema.md` §4.

## Acceptance criteria

Observable in the Vercel preview. One yes/no each.

**Form + write path**

- `entries/test-log.ts` registered; route `app/(authed)/entries/test/new/page.tsx`. `defaultEntryState: 'draft'`.
- `test_type` radio selects mode; the data-entry area uses the new **`raw-data-table`** block (paste-friendly — accepts tab/comma-separated paste).
- Typed columns written: `test_label`, `test_date`, `test_type` (CHECK `pass_fail`/`single_measure`/`custom`), optional `robot_version_hw_id`. `extras`: `hypothesis`, `field_setup`, `method_steps`, `raw_rows`, `custom_columns` (custom mode), `computed`, plus the depth text fields per `03-test-log.md` §7.
- Can file all three: a pass/fail Test Log, a single-measure Test Log, and a custom-table Test Log. Each appears in `/entries/list` and opens on the detail page.

**Auto-compute (the core)**

- `lib/compute/test-stats.ts` — **pure, unit-tested** functions (vitest; Phase 2's named safety net per `00-plan.md` §3 / `CLAUDE.md` rule 6). Called server-side on submit **and** wired into the fallback importer path (shared module — path-independent compute, no DB trigger).
- pass/fail → N, successes, pass rate, 95% Wald CI, failure-mode tally if a note/category column present.
- single measure → N, mean, sample stddev, min, max, 95% CI of the mean.
- custom → per-column by kind: `number`→mean/stddev/min/max, `pass_fail`→rate+CI, `category`→frequency counts, `text`→shown not summarized; if a category column is marked a "condition", numeric stats also computed per group.
- Results stored in `extras.computed` and surfaced via **`computed-readonly`** blocks on the form/detail page — the tester never types a statistic.

**Time-series delta**

- On submit, look up the most recent prior `test_logs` row with the same `test_label`; compute and store the delta on the headline stat.
- Create the **`test_series`** rollup table (migration — approval-required) and write a row per submit (`test_label`, `test_log_id`, `test_date`, `headline_stat`, `headline_label`). Filing a second run with the same label shows the delta in the UI.

**Regression**

- `pnpm verify` + both CI checks green. Tier 1 entries and the 2B forms still file.

## Out of scope

- **AI analysis fallback** (`03-test-log.md` §5 — the "Analyze this data" action, `extras.ai_analysis`). Code-first only this batch; AI is a separate bounded follow-up (own item, server-side, daily/entry cap). Do **not** wire any Claude call in 2C.
- **Comp Recap companion view** that reads the Test Log trend — that's **2D** (consumes `test_series` produced here).
- **Decision Log, Comp Recap forms** — later batches.
- **`updateEntry` full flow** — depth fields are fillable-later, but if the draft→complete edit flow isn't yet built, ship 2C with the depth fields editable in the initial form and defer fill-later to the batch that builds `updateEntry` (2E). Flag which you chose in the PR.

## Open questions

1. **`test_series` table vs derive-on-read.** `01-schema.md` §4 leaves this to 2C: a rollup table for cheap last-run/trend lookups, or query `test_logs` by label on read. Recommend the table (the dashboard and 2D companion view both read trends often) — but if read volume is trivial, deriving on read avoids a second write. Pick one; the acceptance criteria assume the table.
2. **`raw-data-table` paste parsing.** How forgiving should the paste parser be (tab vs comma detection, ragged rows, header row inference)? Keep it minimal — detect delimiter, trim, require the tester to confirm column mapping. Don't over-build.
3. **Headline stat for custom mode.** Which column is the "headline" for the delta when the shape is custom? `03-test-log.md` §4 says "tester-chosen headline column" — add a small selector, defaulting to the first `number` column.

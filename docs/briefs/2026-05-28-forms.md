## What we're building

The core capture surface of MD: a generic, definition-driven form renderer plus the three Phase 1 Tier 1 entry types — Session Log, Outreach Log, Meeting Notes — and a unified list view across them. A team member signs in, picks an entry type, fills a form built from a single TypeScript `EntryDefinition`, and on submit lands in `/entries/list` with their new row at the top. One renderer drives every form. Adding a Phase 2 entry type becomes a ~5-file change (definition + page + registry update), not a new form build.

## Why

This is what makes the app a capture tool instead of a logged-in landing page. Definition of Done in `docs/phase1/00-plan.md` requires all three Tier 1 entry types capturing-and-listing end to end. The architecture (one renderer + many definitions) is also what unblocks Phase 2 — the seven additional Tier 2 entry types reuse this surface verbatim.

## Acceptance criteria

Observable in the browser. ≥1 criterion per task in the batch.

**T13 — Field block system + generic renderer**

- Each of the ten block types (text, long-text, single-select, multi-select, date, number, person-attribution, story-block, action-items, specialty-triggers) renders without error from a `FieldBlock` config object.
- The renderer wires submit to a server action; on success the page redirects to `/entries/list`; on validation failure, red inline errors appear beneath each affected field; on insert failure, a banner shows the form-level error.
- `visibleWhen` with `equalsOptionValue` correctly hides/shows fields based on the current value of another field; hidden fields' values are excluded from the submit payload.
- The "Add new…" affordance in single/multi-select opens an inline popover, posts to a server action, and the new `option_lists` row is auto-selected when the popover closes.

**T14 — Session Log entry + page**

- Navigating to `/entries/sessions/new` (from `/entries/list`) loads the Session Log form per `03-forms.md` §13.
- Filling required fields and submitting inserts a `session_logs` row; columns vs `extras` JSONB split matches the definition's `storage` declarations.
- Filling at least one specialty-triggers checkbox with owner + subject lands the row in `extras.specialty_entries` as an array.
- Submitting with `what_worked_on` blank shows "Required" beneath that field; no insert happens.

**T15 — Outreach Log entry + page**

- Form at `/entries/outreach/new` renders all 22 fields per `03-forms.md` §14.
- Engagement-depth field accepts submission when ≥1 checkbox OR a non-empty custom note is present; rejects with a Zod error when both are empty.
- Three stories required; "Remove" button on the third story is disabled.
- Setting `follow_up_type` to `individual` reveals the "Names for individual follow-up" textarea; setting it to `re-engagement` reveals the "Link to prior Outreach Log" textarea; other values hide both. UUID resolution happens via `equalsOptionValue` per `03-forms.md` §14 follow-up note.
- Submitting valid data inserts an `outreach_logs` row with typed columns populated and `extras.stories`, `extras.engagement_depth`, `extras.specialty_entries` correctly shaped.

**T16 — Meeting Notes entry + page**

- Form at `/entries/meetings/new` renders per `03-forms.md` §15.
- Submitting with zero attendees shows a Zod error ("at least 1 required"); submitting with one or more attendees inserts a `meeting_notes` row.
- Action items are optional; the form accepts a submission with none.

**T17 — Cross-entry list view**

- `/entries/list` shows every active entry from `session_logs`, `outreach_logs`, `meeting_notes` sorted by `created_at` DESC.
- Each row renders: a colored type pill (one color per entry type), the date, the `listSummary` headline (≤100 chars), and the filer's email.
- Empty state ("No entries yet — file one above") renders when zero entries exist.
- Each row is clickable and navigates to a placeholder `/entries/<type>/<id>` page that just renders "Detail view: Phase 2."

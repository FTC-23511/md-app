# Feature: 3C — 24h edit lock + edit_reason audit

## What we're building

The user-facing half of the 24h edit lock. RLS (3B) already refuses out-of-window edits at the database; this batch makes `lib/update-entry.ts` explain *why* in plain language, and lets a Captain/Deputy override an entry older than 24h by supplying an `edit_reason` that is recorded to an `entry_edit_audit` table.

## Why

The 24h lock freezes the contemporaneous record (charter §6) — but a hard RLS refusal currently surfaces as a raw Postgres error (R3). Non-technical users need a friendly "this is locked" message, and overrides need an audit trail so the frozen record stays trustworthy.

## Acceptance criteria

- Migration applies on dev then prod: `entry_edit_audit(id, entry_type, entry_id, editor_member_id, edit_reason, edited_at)` exists with RLS (INSERT for Captain/Deputy, SELECT for active members) + the `record_entry_edit(...)` `SECURITY DEFINER` RPC.
- A **general member** opening their own entry older than 24h sees a friendly "This entry is locked — only the Captain or Deputy can edit it after 24 hours" message; a younger own-entry is editable.
- A **Captain** editing a >24h entry they didn't author is shown an `edit_reason` field; saving without it shows a reason-required message; saving with it succeeds and writes exactly one `entry_edit_audit` row.
- Mentor/inactive users get "Your role does not permit editing entries."
- On any unexpected permission failure, a generic "You don't have permission to make this change" appears instead of a raw error (RLS stays the backstop).
- `pnpm verify` green.

## Out of scope

- Changing the RLS predicate itself (done in 3B; this batch only mirrors it in TS for messaging).
- An audit-log viewer UI — the row is written; surfacing it is later.
- Outreach Reporter / Subsystem Lead extended-edit UI affordances beyond the existing edit flow (RLS already permits them; messaging parity is enough).

## Open questions

- **Where the `edit_reason` input lives** — inline in the existing detail-page edit/fill form, shown only when the lock + override applies. Confirm at PR.
- **RPC atomicity** — wrap only the audit INSERT in the RPC (simpler; RLS already authorized the row) vs wrap the column UPDATE too (full atomicity). Default: audit-only RPC.

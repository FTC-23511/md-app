# Feature: 2F — Media to Google Drive

<!-- docs/briefs/2026-06-15-2f-media.md — Phase 2 step 2F. Spec: docs/phase2/04-media.md + 01-schema.md §7. -->

## What we're building

A `media-links` field block usable on any entry. Paste a link or upload a file;
team-owned/fragile media (uploads, Discord attachments, loose image/video links)
is **automatically ingested** server-side into the team Google **Shared Drive** via
a service account, and the DB stores only the Drive link + a thumbnail. YouTube and
Vimeo links **pass through** unchanged (stay native). A preview shows on the entry's
detail page. DB storage stays near zero.

## Why

Resolves the App Charter §4/§8 "photos exceed free tier" risk and removes the manual
Drive-upload + manual-sharing friction. Shared-Drive ownership means media survives
student turnover (a Sustain-Award benefit). Last unbuilt media capability in Phase 2.

## Acceptance criteria

- `media-links` block renders repeating rows of `{url_or_file, caption, permission_status, role?}`; wire convention `name__url`, `name__caption`, etc. (matches existing composite blocks).
- Pasting a **YouTube/Vimeo** link stores it as-is: `provider` = youtube/vimeo, `ingest_status` = `not_needed`, native thumbnail.
- Uploading a file **or** pasting a Discord/Imgur/direct image-video link triggers **ingest**: app fetches bytes server-side → uploads to `MD-media/<entry_type>/` in the Shared Drive → sets link-sharing to "anyone with the link can view" → stores Drive link + `drive_file_id` + `provider='google_drive'` + `source_provider` + `media_type` + thumbnail in `media_links`. `ingest_status` flows `pending`→`done`/`failed`.
- Oversized uploads are **rejected** with a clear message ("too big — upload to YouTube and paste the link"), reinforcing passthrough. (Threshold chosen to stay under the Vercel serverless body/time limit — default ~4 MB; set in the brief's open question.)
- A pre-existing Drive share link pasted directly is stored as-is (`provider='google_drive'`, `ingest_status='not_needed'`); if not link-viewable the preview fails and the user is warned to fix sharing.
- Entry **detail page** renders attached media: image thumbnail/embed, video embed for YouTube/Vimeo, caption, permission badge.
- Retrofit: Hardware Change Log photos and Session/Outreach photos can attach via `media-links`. (Outreach `story-block.photo_url` stays as-is per App Lead — not migrated.)
- `pnpm verify` green; manual smoke against a live Shared Drive once secrets are set.

## Out of scope

- Migrating existing `story-block.photo_url` data into `media_links` (App Lead: leave as-is).
- The nightly link-health sweep job itself — only the `(last_checked_ok)` index + columns need to exist this round; the cron job can be a follow-up (it's a `MD_App_Charter.md` §9 / Friday-15 item, not gating 2F's DoD).
- Any per-media access control beyond Drive link-sharing (Phase 3 RLS).

## Open questions

- **Oversized-file threshold.** Default **4 MB** (safe under Vercel hobby serverless limits). Bump to 10 MB only if the deploy tier allows — confirm at PR. Anything larger → passthrough message.
- **`googleapis` dependency.** 2F needs the Google Drive SDK (`googleapis`), which is a **new dependency** — CLAUDE.md rule 2 requires App Lead approval at PR. No lighter in-stack option (raw REST + service-account JWT signing is more code and more risk). **Approve at PR.**
- **media_links indexes.** The table shipped in 2A (#34) per `01-schema.md` §7, which specifies the `(entry_type, entry_id)` and `(last_checked_ok)` indexes — verify they exist; add a small migration only if missing. Service-role grants for the ingest write path likely covered by `20260602000001_grants_for_service_role.sql` — verify.
- **Env var names.** `GOOGLE_SERVICE_ACCOUNT_KEY` (JSON) + `GOOGLE_DRIVE_MEDIA_FOLDER_ID`, server-only (never `NEXT_PUBLIC_*`). Confirm naming at PR.

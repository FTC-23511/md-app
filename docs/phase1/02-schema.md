# Database schema

This document is the canonical specification for every table, column, constraint, index, and policy in the Phase 1 schema. It is also the reference Claude Code consults when a Phase 2+ task requires adding columns, tables, or indexes.

Sections 0–2 establish conventions. Sections 3–6 are the four migration files in execution order. Sections 7–9 document patterns and workflows.

## 0. Overview

Phase 1 ships **four migration files**, in this order:

| File (suffix)                  | Contents                                                                                                                   |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `001_functions_and_types.sql`  | Trigger functions, shared CHECK constraint values                                                                          |
| `002_supporting_tables.sql`    | `members` (with auth trigger), `option_lists` (with seed data), `flags`, `classification_index`, `award_criteria_snapshot` |
| `003_entry_tables.sql`         | All 10 entry-type tables + `test_trials` child table                                                                       |
| `004_indexes_and_policies.sql` | Secondary indexes + RLS enablement + permissive Phase 1 policies                                                           |

Filenames are generated by `supabase migration new <name>`, which prepends a timestamp prefix. The actual filenames will look like `20260522123045_functions_and_types.sql`. The doc refers to them by suffix.

## 1. Workflow

The migration workflow per Supabase's documented best practice:

1. From the repo root, run `supabase migration new <name>` to create a new empty migration file.
2. Open the generated file at `supabase/migrations/<timestamp>_<name>.sql` and write the SQL.
3. Run `supabase db reset` locally to drop and rebuild the local database from scratch using all migrations. This verifies the new migration is consistent with what came before.
4. Run `supabase db push` to apply the new migration to the remote project on supabase.com.
5. Commit the migration file to git. Open a PR per the conventions in `01-conventions.md`.

**The golden rule.** Never edit tables in the Supabase web dashboard. All schema changes go through migration files. Dashboard edits bypass the migration history and cause `db push` to fail with sync errors. This rule is named in `CLAUDE.md`'s absolute rules; it applies forever, not just Phase 1.

## 2. Schema-wide conventions

These conventions apply to every table in the schema. They are why the table definitions in §§4–5 are as compact as they are — most of the boilerplate is shared.

### 2.1 Common columns

Every table has these columns:

| Column       | Type          | Default             | Purpose                           |
| ------------ | ------------- | ------------------- | --------------------------------- |
| `id`         | `uuid`        | `gen_random_uuid()` | Primary key                       |
| `created_at` | `timestamptz` | `now()`             | Record creation timestamp         |
| `updated_at` | `timestamptz` | `now()`             | Auto-maintained by trigger        |
| `deleted_at` | `timestamptz` | NULL                | Soft-delete marker; NULL = active |

Every **entry table** additionally has:

| Column        | Type    | Default       | Purpose                                                                                                    |
| ------------- | ------- | ------------- | ---------------------------------------------------------------------------------------------------------- |
| `created_by`  | `uuid`  | NULL          | FK to `auth.users(id)` ON DELETE SET NULL; populated by the server action with the authenticated user's ID |
| `created_via` | `text`  | `'app'`       | Provenance tag: `'app'`, `'fallback_form'`, or `'import'` (CHECK constraint)                               |
| `extras`      | `jsonb` | `'{}'::jsonb` | Free-form storage for non-queried fields                                                                   |
| `entry_state` | `text`  | `'complete'`  | `'draft'` or `'complete'` per charter Draft → Complete model                                               |

`gen_random_uuid()` is built into PostgreSQL 13+ (no extension needed). Supabase runs PG 15+.

`timestamptz` (timestamp with time zone) is preferred over `timestamp` everywhere — it stores absolute moments unambiguously and converts to the user's locale on display.

### 2.2 The `extras` JSONB column

Every entry table has an `extras jsonb NOT NULL DEFAULT '{}'::jsonb` column. Fields that are _stored and retrieved as a whole, never the basis of a query_, live in `extras` as JSON keys. Fields that are _queried, filtered, joined, indexed, or constrained_ live as typed columns.

Examples drawn from the actual schema:

- `outreach_logs.event_date` is a typed column — list view filters and sorts by date.
- `outreach_logs.event_type_option_id` is a typed FK column — analytics queries group by event type.
- `outreach_logs.extras.stories` is a JSONB array of `{person, happened, quote, permission}` objects — displayed on the entry detail page, never filtered on at the SQL level.
- `outreach_logs.extras.engagement_custom_note` is a JSONB string — narrative texture, no queries.

The §8 section walks through the full rule with more examples.

### 2.3 Soft delete

Hard-deleting an entry destroys a contemporaneous record. The Charter is explicit that the contemporaneous record is the basis of credibility — losing it is the opposite of what the system exists to do.

So: every table has a `deleted_at timestamptz NULL` column. Deletion is `UPDATE entry_table SET deleted_at = now() WHERE id = $1`. Every read query filters with `WHERE deleted_at IS NULL`. A partial index on each table makes the filter cheap (§6).

Soft-deleted rows are still in the table. They can be restored with `UPDATE ... SET deleted_at = NULL`. Forever-purging of rows is a Phase 4+ concern (data retention policy), not Phase 1.

### 2.4 The `updated_at` trigger

A shared trigger function updates `updated_at` to `now()` on every UPDATE. Defined once in migration 001; applied per-table in migrations 002 and 003.

### 2.5 RLS enablement

Every table has Row Level Security enabled in migration 004 with one **permissive Phase 1 policy**: any authenticated user can do anything. Phase 3 replaces these permissive policies with per-role policies.

This is good hygiene (Supabase advisor flags tables without RLS as a security issue) and belt-and-suspenders with the middleware email allowlist: middleware says "only the allowlisted email gets through," RLS says "you must be authenticated to touch any row." Two layers, both pass before any read or write.

## 3. Migration 001: functions and types

```sql
-- =========================================================================
-- Migration 001: functions_and_types
-- Purpose: Shared trigger functions and any common SQL helpers used by
-- subsequent migrations.
-- =========================================================================

-- The updated_at trigger function. Applied via CREATE TRIGGER in later
-- migrations. Sets NEW.updated_at to the current transaction timestamp
-- before the UPDATE writes.
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

That's the whole file. Migration 001 is intentionally small — it establishes one shared helper and nothing else. The auth trigger function lives in migration 002 because it depends on the `members` table.

## 4. Migration 002: supporting tables

Order matters: `members` first (because the auth trigger depends on it), then `option_lists` and its seed data, then `flags`, `classification_index`, and `award_criteria_snapshot`.

### 4.1 `members`

Mirrors `auth.users` with display-friendly fields. Populated by an auth trigger so the application doesn't have to call insert-on-signup explicitly.

```sql
CREATE TABLE members (
  id            uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email         text NOT NULL UNIQUE,
  display_name  text NOT NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  deleted_at    timestamptz
);

CREATE TRIGGER members_set_updated_at
BEFORE UPDATE ON members
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- Auth trigger: create a members row whenever a new auth.users row is
-- inserted. The display_name defaults to the part of the email before @;
-- the user can change it later via a profile page (Phase 2+).
CREATE OR REPLACE FUNCTION handle_new_auth_user() RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.members (id, email, display_name)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NULLIF(split_part(NEW.email, '@', 1), ''), 'Member')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
AFTER INSERT ON auth.users
FOR EACH ROW EXECUTE FUNCTION handle_new_auth_user();
```

`ON DELETE CASCADE` on the FK to `auth.users` means: deleting a user in `auth.users` removes their `members` row. Their entries (which FK back via `created_by` with `ON DELETE SET NULL`) survive with `created_by = NULL`.

The `SECURITY DEFINER` on the trigger function lets it write to `public.members` even when invoked from the `auth` schema's INSERT path. This is the standard Supabase pattern for profile-mirroring triggers.

### 4.2 `option_lists`

The lookup table for all user-addable option groups. Single shared table; the `category` column discriminates between groups.

```sql
CREATE TABLE option_lists (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  category      text NOT NULL CHECK (category IN (
                    'event_type',
                    'engagement_depth',
                    'follow_up_type',
                    'our_role',
                    'meeting_type',
                    'subsystem',
                    'relationship_type',
                    'relationship_status',
                    'change_type'
                  )),
  value         text NOT NULL,    -- canonical machine-readable (e.g. 'public-showcase')
  label         text NOT NULL,    -- human-readable (e.g. 'Public showcase')
  sort_order    integer NOT NULL DEFAULT 0,
  is_seed       boolean NOT NULL DEFAULT false,
  created_by    uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  deleted_at    timestamptz
);

CREATE TRIGGER option_lists_set_updated_at
BEFORE UPDATE ON option_lists
FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- Partial unique index: a (category, value) combination is unique among
-- active (non-deleted) rows. Soft-deleted rows don't conflict with new ones.
CREATE UNIQUE INDEX option_lists_category_value_active
ON option_lists (category, value)
WHERE deleted_at IS NULL;
```

The `is_seed` boolean distinguishes charter-defined options (inserted by this migration) from user-added options (inserted at runtime via the form's "Add new..." affordance). It's not used for access control — anyone authenticated can edit any option in Phase 1 — but it's useful for showing seed vs. custom in the UI and for the Phase 5+ builder.

### 4.4 `flags`

Tracks Tier 1 → Tier 2 flagging per Charter §11. One row per flag.

```sql
CREATE TABLE flags (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  parent_entry_type   text NOT NULL,  -- e.g. 'session_log', 'outreach_log'
  parent_entry_id     uuid NOT NULL,
  target_entry_type   text NOT NULL CHECK (target_entry_type IN (
                          'decision_log', 'hw_change_log', 'sw_change_log',
                          'test_log', 'contact_log'
                        )),
  owner_member_id     uuid REFERENCES members(id) ON DELETE SET NULL,
  subject             text NOT NULL,
  status              text NOT NULL DEFAULT 'open'
                        CHECK (status IN ('open', 'closed', 'cancelled')),
  resolved_entry_id   uuid,  -- populated when status = 'closed'; FK is informal
                             -- because it could point to any entry table
  opened_at           timestamptz NOT NULL DEFAULT now(),
  closed_at           timestamptz,
  created_at          timestamptz NOT NULL DEFAULT now(),
  updated_at          timestamptz NOT NULL DEFAULT now(),
  deleted_at          timestamptz
);

CREATE TRIGGER flags_set_updated_at
BEFORE UPDATE ON flags
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

The `parent_entry_type / parent_entry_id` pair points at any entry table — this is a polymorphic association, not a true FK. We could enforce integrity via triggers, but in Phase 1 we trust the application layer to write valid references (the form writes the parent's type and id when it creates the flag).

`resolved_entry_id` similarly points at the eventual Tier 2 entry that closed this flag; informally a FK, formally a uuid column.

## 5. Migration 003: entry tables

Each of the 10 entry tables follows the conventions in §2. Typed columns hold queried/filtered fields; `extras` JSONB holds everything else. The split per table is the same one I outlined in the plan, now in full SQL.

Each entry table gets a `set_updated_at` trigger application after creation. The trigger statement is identical for every table; I'm showing it once explicitly on `session_logs` and abbreviating with a comment on the rest.

### 5.1 `session_logs`

```sql

  -- common columns
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now(),
  deleted_at      timestamptz,
  created_by      uuid REFERENCES auth.users(id) ON DELETE SET NULL,
```

### 5.9 `test_trials` (child of `test_logs`)

```sql
CREATE TABLE test_trials (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  test_log_id    uuid NOT NULL REFERENCES test_logs(id) ON DELETE CASCADE,
  trial_number   integer NOT NULL,
  outcome        text,                          -- 'success' | 'fail' (for pass-fail tests)
  measurement    numeric,                       -- numeric value (for continuous tests)
  notes          text,
  created_at     timestamptz NOT NULL DEFAULT now(),
  CHECK ((outcome IS NOT NULL AND measurement IS NULL)
      OR (outcome IS NULL AND measurement IS NOT NULL)),
  UNIQUE (test_log_id, trial_number)
);
```

One row per trial. The CHECK constraint ensures exactly one of `outcome` and `measurement` is populated (matching the parent's `test_type`). `ON DELETE CASCADE` is intentional: if a Test Log is hard-deleted (rare), trials go with it. Soft-deletes leave trials alone — the test_logs row's `deleted_at` is the canonical signal.

Stats on the parent `test_logs` row (pass*rate, ci*\*, mean, stddev, etc.) are computed by the application's insert helper when trials are submitted. Phase 2 implements the compute; Phase 1 leaves the columns nullable.

## 7. The `option_lists` pattern

How forms read the table:

```typescript
// Server-side fetch at page load
const { data: eventTypes } = await supabase
  .from('option_lists')
  .select('id, value, label')
  .eq('category', 'event_type')
  .is('deleted_at', null)
  .order('sort_order')
  .order('label');
```

These rows populate the dropdown options. The form stores the selected `id` as the `event_type_option_id` FK on the entry row.

**Adding a new option from the form's "Add new..." affordance:**

```typescript
// Server action when user submits "Add new..." with their custom label
const value = slugify(userLabel); // e.g. "Local hospital demo" → "local-hospital-demo"
const { data: newOption } = await supabase
  .from('option_lists')
  .insert({
    category: 'event_type',
    value,
    label: userLabel,
    is_seed: false,
    created_by: currentUser.id,
  })
  .select()
  .single();
// Now newOption.id can be used as the event_type_option_id on the entry being filed
```

If the slugified value collides with an existing active option (Postgres returns a unique-constraint error), the form surfaces "That option already exists" and lets the user pick the existing one.

**The "Other..." pattern (for the `'other'` seed value).** Several seed lists include an `'other'` option as the last entry. This is a curated catch-all for things that don't fit. The "Add new..." affordance is a separate path: it creates a new specific option for reuse, while `'other'` is a single bucket for one-offs.

## 8. The column-vs-`extras` rule

The rule from `01-conventions.md` §5 in practice:

| If the field is…                                            | Then…          | Example                                                                      |
| ----------------------------------------------------------- | -------------- | ---------------------------------------------------------------------------- |
| Filtered, sorted, or grouped in queries                     | Typed column   | `outreach_logs.event_date` (sorted in list view)                             |
| Joined on or constrained as a FK                            | Typed column   | `decision_logs.parent_session_id`                                            |
| Used in a unique constraint or index                        | Typed column   | `outreach_logs.event_type_option_id` (FK + indexed)                          |
| Used in an analytics rollup                                 | Typed column   | `outreach_logs.total_attendees` (rolled up in KPI dashboard)                 |
| Stored and retrieved as a whole, never the basis of a query | `extras` JSONB | `outreach_logs.extras.stories` (displayed only)                              |
| Variable in shape across rows                               | `extras` JSONB | `decision_logs.extras.weighted_matrix` (only present when the trigger fires) |
| A bag of arbitrary key-value pairs                          | `extras` JSONB | `comp_recaps.extras.judging.pit_panels[*].likely_awards[]`                   |

When you're unsure, default to `extras`. Promoting a field from `extras` to a typed column later is a small migration; demoting a typed column to `extras` is a larger one and you lose existing query infrastructure.

## 9. Adding new columns, tables, or option groups

When a Phase 2+ task requires schema changes:

### Adding a new typed column to an existing table

1. `supabase migration new add_<column>_to_<table>`
2. Write `ALTER TABLE <table> ADD COLUMN <column> <type> [DEFAULT <default>];`
3. If the column is queried, add an index in the same migration.
4. `supabase db reset` to verify.
5. `supabase db push`.
6. Update the entry definition in `src/entries/<type>.ts`.

### Promoting a field from `extras` to a typed column

1. Add the typed column with a nullable type.
2. Backfill: `UPDATE <table> SET <column> = (extras->>'<key>')::<type> WHERE extras ? '<key>';`
3. Optionally make NOT NULL once backfilled, in a separate migration.
4. Optionally remove the `extras` key in a follow-up migration: `UPDATE <table> SET extras = extras - '<key>';`

### Adding a new option group (new category in `option_lists`)

1. `supabase migration new add_<category>_to_option_lists_category_check`
2. Drop and recreate the CHECK constraint on `option_lists.category` to include the new value.
3. Insert seed rows for the new category if any.
4. Add the FK column on the consuming entry table (separate migration if substantive).

### Adding a new entry table

1. `supabase migration new add_<entry>_table`
2. Write the `CREATE TABLE` following the conventions in §2 (common columns + entry-specific + extras + entry_state).
3. Add the `set_updated_at` trigger.
4. Add the RLS enable + permissive policy.
5. Add indexes for created_by, created_at DESC, and any FKs in `004` (or a new indexes migration if many are needed).
6. Create the entry definition in `src/entries/<type>.ts`.
7. Add to `src/entries/_registry.ts`.

Never edit an existing migration file after it's been applied to the remote. Always write a new migration. Editing applied migrations breaks the migration history table and produces `db push` sync errors.

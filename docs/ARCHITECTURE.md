## The 10 entry types as a hybrid schema

The schema uses a single `entries` table for cross-cutting fields (team_id,
type, author, date, state, showcase flag, overlay) plus a 1:1 detail table
per entry type for type-specific columns.

```
entries (id, type, team_id, author_id, ...)
   │
   ├── session_logs        (entry_id PK/FK)
   ├── outreach_logs       (entry_id PK/FK)  ──> outreach_stories (1:N)
   ├── meeting_notes       (entry_id PK/FK)
   ├── competition_recaps  (entry_id PK/FK)  ──> pit_panels, failure_analyses,
   │                                            notable_matches, strategic_insights
   ├── decision_logs       (entry_id PK/FK)  ──> decision_alternatives
   ├── hardware_change_logs (entry_id PK/FK) ──> hardware_change_deltas
   ├── software_change_logs (entry_id PK/FK)
   ├── test_logs           (entry_id PK/FK)  ──> test_series (N:1)
   ├── contact_logs        (entry_id PK/FK)  ──> contacts (N:1)
   └── subsystem_handoffs  (entry_id PK/FK)
```

Cross-cutting tables also key off entries:

- `entry_revisions` — audit trail of every UPDATE (trigger snapshots OLD row)
- `entry_attendees` — who was present
- `media_attachments` — photos/videos/voice memos
- `flags` — Tier-1-triggered Tier-2 to-do queue
- `action_items` — meeting outputs
- `classification_tags` — award-criterion index (Appendix A weekly pass)

## RLS access model

Every table has RLS enabled. The base rules:

- **Read**: any member of the team can read any of their team's data.
- **Insert**: any writing role (captain / deputy / lead / general) can insert.
- **Update author-own**: within 24h of creating an entry, the author can edit it.
- **Update captain/deputy**: any time, on any entry (this is the "edit with
  reason" path — the app passes the reason via `entries.last_edit_reason`).
- **Delete**: captain only.
- **Public showcase**: anonymous users can SELECT entries where
  `is_public_showcase = TRUE` (only — no joins to other tables).

Detail tables (session_logs, etc.) delegate to the parent entry's permissions
via the `can_read_entry()` and `can_write_entry()` helpers — see migration 006.

## Why these choices?

A few decisions you'll meet often, with rationale:

- **Supabase, not raw Postgres + custom auth.** Auth, Storage, and RLS in one
  managed service. We use ~10% of what Supabase offers; that 10% would be
  weeks of work to build ourselves.
- **Server Components by default.** Pages fetch from Supabase directly on the
  server; the browser gets HTML with data already in it. Only forms and
  interactive widgets are Client Components.
- **Zod schemas as the single source of truth for inputs.** Same schema
  validates the form, the server action, and types the data — no drift
  possible.
- **Generated types from the DB.** When the schema changes, run `pnpm db:types`
  and TypeScript will tell you everywhere code now disagrees with reality.
- **Strict TypeScript including `noUncheckedIndexedAccess`.** Indexing into an
  array returns `T | undefined`. This catches "what if the row isn't there?"
  bugs at compile time.
- **Magic-link auth, not passwords.** No password reset, no leak risk, no
  forgotten passwords. Email is enough for a 30-person team.
- **One repo, one app.** Monorepo overhead isn't worth it.
